# Network Design

## 1. 概要

本ドキュメントでは、C++とDirectX12を用いた自作エンジン上で開発しているリアルタイム映像通信基盤の設計をまとめます。

本システムは、対戦ゲームにおいて離れた相手の映像をゲーム画面上に低遅延で表示し、臨場感のある体験を実現することを目的としています。

単に映像を送受信するだけでなく、RTT、ジッタ、パケットロス、フレーム欠落、ビットレートなどを計測・可視化し、通信状態を評価しながら安定性を改善できる仕組みを重視しています。

---

## 2. 開発目的

リアルタイム映像通信では、遅延を小さくするだけでなく、ジッタやパケット欠落による不安定さを抑えることが重要です。

特にゲーム用途では、映像の遅れや乱れがプレイヤーの体験に直接影響します。そのため、本プロジェクトでは以下を目標としています。

- 低遅延で映像を送受信する
- UDP通信で発生する欠落や順序ずれに対応する
- 受信側でフレームを安定して再構成する
- 通信状態をログやUIで観測できるようにする
- 測定結果をもとに改善前後を比較できるようにする

---

## 3. 全体構成

```text
Camera
  ↓
Media Foundation Capture
  ↓
JPEG Encode
  ↓
RNVP Packetize
  ↓
UDP Send
  ↓
UDP Receive
  ↓
Frame Reassembler
  ↓
Jitter Buffer
  ↓
Decode / Texture Upload
  ↓
DirectX12 Rendering
```

送信側では、Media Foundationで取得したカメラ映像をJPEG形式に圧縮し、RNVPプロトコルでチャンク単位に分割してUDP送信します。

受信側では、RNVPヘッダをもとにチャンクを管理し、必要なデータが揃ったフレームだけを再構成します。再構成されたフレームはジッタバッファを経由し、表示側へ渡されます。

---

## 4. RNVPプロトコル

RNVPは、UDP上でリアルタイム映像フレームを扱うために定義した独自プロトコルです。

UDPは低遅延である一方、パケット欠落や順序ずれが発生します。そのため、大きな映像フレームを複数のチャンクに分割し、各パケットに識別情報を付与することで、受信側でフレーム単位に再構成できるようにしています。

### 主なヘッダ情報

| 項目 | 役割 |
|---|---|
| MagicID | RNVPパケットであることを識別する値 |
| frameId | 映像フレームを識別するID |
| chunkIndex | フレーム内のチャンク番号 |
| chunkCount | 1フレームを構成する総チャンク数 |
| sequence | パケット順序や欠落検出に用いる番号 |
| sendTime | RTTや遅延測定に使用する送信時刻 |
| codecType | JPEGなどの映像データ形式を識別する情報 |

### RNVPを導入した理由

RNVPを導入した理由は、受信側で「どのデータが、どのフレームの、どの部分に対応するのか」を判定できるようにするためです。

これにより、以下の処理が可能になります。

- フレーム単位での再構成
- 欠落チャンクの検出
- 重複パケットの検出
- 古いフレームの破棄
- ACKによる不足チャンクの通知
- RTTやジッタなどの測定

---

## 5. 送信側の処理

送信側では、Media Foundationでカメラ映像を取得し、JPEG形式に圧縮したうえでUDP送信します。

### 処理の流れ

```text
1. カメラ映像を取得
2. 映像フレームをJPEG形式に圧縮
3. 1フレームを複数のチャンクに分割
4. 各チャンクにRNVPヘッダを付与
5. UDPで送信
6. ACKや再送要求に応じて必要なチャンクを再送
```

### 送信側で重視している点

- 低遅延で送信すること
- 1パケットが大きくなりすぎないようにチャンク分割すること
- frameIdやchunkIndexを付与し、受信側で再構成しやすくすること
- sendTimeを付与し、RTTなどの測定に利用できるようにすること
- 欠落した場合に必要なチャンクだけを再送できるようにすること

---

## 6. 受信側の処理

受信側では、受け取ったRNVPパケットを解析し、frameIdとchunkIndexをもとにチャンクを管理します。

必要なチャンクが揃った場合のみ、1つの完成フレームとして再構成し、表示側へ渡します。

### 処理の流れ

```text
1. UDPでRNVPパケットを受信
2. MagicIDやヘッダ情報を検証
3. frameIdごとにチャンクを管理
4. chunkIndexをもとにデータを配置
5. 全チャンクが揃ったらフレームを再構成
6. 完成フレームをジッタバッファへ渡す
7. 一定時間内に完成しないフレームは破棄
```

### 受信側で重視している点

- 不正なパケットを表示側に渡さないこと
- 欠落や重複を検出できること
- 未完成フレームをいつまでも保持しないこと
- フレーム順序を管理し、古いフレームを適切に破棄すること
- 通信状態を計測できること

---

## 7. ACKと選択再送

UDPではパケットが欠落する可能性があります。  
そのため、受信側で不足しているチャンクを検出し、ACKによって送信側へ通知する仕組みを導入しています。

### 目的

- 欠落チャンクを把握する
- 必要なチャンクのみを再送する
- フレーム全体を再送せず、通信量を抑える
- 低遅延性と信頼性のバランスを取る

### 処理イメージ

```text
Receiver
  ↓
Missing chunk detected
  ↓
ACK with missing chunk indices
  ↓
Sender
  ↓
Retransmit only missing chunks
```

選択再送により、欠落した部分だけを補うことを目指しています。  
ただし、リアルタイム性を重視するため、古すぎるフレームは再送対象にせず、破棄する設計にしています。

---

## 8. ジッタバッファ

ネットワーク通信では、パケットやフレームの到着間隔に揺らぎが発生します。  
この揺らぎをそのまま描画に反映すると、映像のカクつきや乱れにつながります。

そのため、受信した完成フレームをすぐに表示側へ渡すのではなく、ジッタバッファで一時的に保持し、順序やタイミングを整えてから表示側へ渡します。

### 目的

- 到着間隔の揺らぎを吸収する
- フレーム順序を安定させる
- 古いフレームを破棄する
- 表示側へ渡すタイミングを安定させる

### 処理イメージ

```text
Completed Frame
  ↓
Jitter Buffer
  ↓
Delay / Reorder / Drop
  ↓
Rendering Queue
```

## 9. ImGuiによる監視・操作

本プロジェクトでは、通信状態を確認するためにImGui上にネットワーク監視パネルを実装しています。

### 主な表示項目

| 項目 | 内容 |
|---|---|
| RTT | 往復遅延 |
| Jitter | 到着間隔の揺らぎ |
| Packet Loss | パケット欠落率 |
| Frame Drop | フレーム欠落数 |
| Bitrate | 通信量 |
| Receive FPS | 受信フレームレート |
| Reassembler State | フレーム再構成状態 |
| Jitter Buffer State | バッファ内のフレーム数や遅延状態 |
| ACK / Retransmit | ACK数や再送状態 |

### 主な操作・出力項目

| 項目 | 内容 |
|---|---|
| Reset Stats | 計測値のリセット |
| CSV Output | 測定結果をCSV形式で出力 |
| Markdown Report | 測定結果をMarkdown形式で出力 |
| Buffer Parameters | ジッタバッファなど一部パラメータの調整 |

ImGuiは単なるデバッグ表示ではなく、通信状態を測定しながら改善するための観測ツールとして使用しています。

## 10. ログによる原因切り分け

開発中に最も苦労したのは、受信映像が乱れる問題です。

映像が乱れた際、原因が通信、再構成、デコード、GPU転送、描画のどこにあるのかを判断しにくいという課題がありました。

そこで、以下の各段階にログを追加し、正常なデータがどの段階で崩れているのかを確認しました。

```text
Capture
  ↓
Encode
  ↓
Packetize
  ↓
Send / Receive
  ↓
Reassemble
  ↓
Decode
  ↓
GPU Texture Upload
  ↓
Render
```

### ログで確認した情報

- 映像形式
- フレーム番号
- チャンク番号
- 受信状況
- 欠落・重複の有無
- デコード状態
- GPU転送状態
- 描画状態

この切り分けにより、映像の乱れは、受信データがどのフレームのどの部分に対応するかを正しく管理できていないことが主な原因だと分かりました。

そのため、MagicIDやframeIdを用いて受信データを識別する仕組みを強化し、不正なデータや不完全なフレームを表示側に渡さないように改善しました。

## 11. 測定・評価・改善

本プロジェクトでは、実装して終わりではなく、通信状態を測定し、改善前後を比較できるようにしています。

### 測定項目

- RTT
- ジッタ
- パケットロス
- フレーム欠落
- ビットレート
- 受信FPS
- 再送回数
- バッファ状態

### 出力形式

- ImGuiによるリアルタイム表示
- CSV形式での測定結果出力
- Markdown形式での実験レポート出力

### 改善サイクル

```text
Implementation
  ↓
Measurement
  ↓
Evaluation
  ↓
Improvement
```

通信状態を数値として確認し、改善前後を比較することで、どの変更が安定性に影響したのかを評価できるようにしています。

## 13. 学んだこと

リアルタイム性が求められるシステムでは、単に通信処理を実装するだけでは不十分であり、状態を観測し、原因を切り分けながら改善できる仕組みが重要だと学びました。

また、通信品質を高めるには、実装だけでなく、測定・評価・改善のサイクルを回すことが重要だと実感しました。

この経験を通じて、見えにくいシステムの状態をデータとして捉え、安定性や信頼性を高める開発姿勢を身につけました。
